UnityEngine.Object和System.Object

Created2022-06-29|Updated2022-06-29|Unity
|Post View:

引子

string s = string.Empty;
var go = new GameObject("goodBoy");
        
DestroyImmediate(go);
go = null;
if (!go) s += "A";

if (go is null) s += "B";
        
if (go == null) s += "C";

if ((System.Object)go == null) s += "D";

Debug.Log(s);

最后输出AC。

首先而System.Object的==null是判断是否为空指针，

而UnityEngine.Object的”==”是做过操作符重载的，当然也重载了≠。

UnityEngine.Object的==会调用以下的片段。

private static bool CompareBaseObjects(Object lhs, Object rhs)
{
	//强转为System.object
	//操作符左边的值是否为null
    bool flag = (object)lhs == null;
	//操作符右边的值是否为null
    bool flag2 = (object)rhs == null;

	//都为null 返回true
	//此处说明，如果C#方面的引用已被置为空引用,那游戏对象肯定不存在.
    if (flag2 && flag)
    {
        return true;
    }
						
						 
    if (flag2)
	{
        return !IsNativeObjectAlive(lhs);
    }

    if (flag)
    {
        return !IsNativeObjectAlive(rhs);
    }
					  
	//（通过对GUID和LocalID的计算，得到全局唯一的m_InstanceID）
	// 对比二者的m_InstanceID
    return lhs.m_InstanceID == rhs.m_InstanceID;
}

另一段是

private static bool IsNativeObjectAlive(Object o)
{
if (o.GetCachedPtr() != IntPtr.Zero)
{
    return true;
}
//判断对象是否派生自MonoBehaviour 或者 ScriptableObject
if (o is MonoBehaviour || o is ScriptableObject)
{
    return false;
}
//判断InstanceID还存不存在
return DoesObjectWithInstanceIDExist(o.GetInstanceID());

}

得出结论：对于被Destory掉了的UnityEngine.Object对象（底层C++对象），上面挂载的代码已经不会继续执行，C#方面的引用变量是否置为空引用已经无关紧要。

所以Unity重写了UnityEngine.Object的判空，只判断UnityEngine.Object是否被销毁（注意这个销毁让UnityEngine.Object不能继续执行它上面的代码），而不是C#引用是否被置为null。

题外话：像是？和？？都是跟System.Object靠边的，慎用在UnityEngine.Object上。

对于UnityEngine.Object被销毁后，C#层面的引用几时会置为空引用和实例几时被GC掉，这个有待考据。
